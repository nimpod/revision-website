Introduction to the Data-Link Layer

Layer 2 - The Data-Link Layer detects damaged or lost frames and recovers from the problem.

The two main features of the Data-Link Layer are:

  1. To detect damage/loss of frames and recover
  2. To never let a damaged frame pass through the network

The bits we talked about in the Physical layer are of no interest to us anymore. Instead they are grouped into frames. The frame is now the transmitted element, in the form of a block of bytes.

Put more formally, "The transmitted element in the Data-Link Layer is called a frame, which is regarded a block of bytes".


The three main responsibilities of the data-link layer are:

  1. Framing - delimiting data units flowing on the channel.
  2. Error control - detecting and correcting errors introduced from the physical layer (see section 4.2 on why frames go wrong)
  3. Flow control - prevents a fast sender from out-pacing a slow receiver

The Data-Link Layer provides two services:

  1. connection-orientated services
  2. connectionless services

When frames go wrong...

Frames can be lost in many ways. Here are four examples:

  1. Short periods of disconnection - disconnection by a few milliseconds is enough to loose the entire frame.
  2. Address of frame is damaged
  3. Software bugs
  4. Computer crash

Lot's of things can go wrong when sending frames and acknowledgements over a network. Here are 6 sequential scenarios in the form of questions:


Every acronym that has been mentioned in a lecture
# Question Answer
Q1 Suppose sender(A) sends a frame to receiver(B) How will B be able to tell that the frame has been damaged during transmission?
  • A has to append error check bits to the frame.
  • These error check bits allow B to calculate whether the frame is correct or not.
Q2 How will sender(A) know that receiver(B) has received the frame correctly
  • Along with the error check bits, B must also send some sort of acknowledgement to A.
  • -> ACK indicates correct arrival of the frame
  • -> NAK indicates frame corruption.
Q3 What happens if sender(A) sends a frame to receiver(B), and the frame disappears during transfer? (in other words, we loose the frame...)
  • Deadlock...
  • Deadlock happens because B is waiting for a frame to arrive, but A is waiting for ACK or NAK.
Q4 How can we avoid deadlock?
  • A must use a timer.
  • Make A wait a limited amount of time (we call this the timeout interval)
  • if (timeout interval completes without arrival of an ACK) --> assume frame is lost --> re-transmit frame
Q5 What happens if we loose an ACK?
  • We recover using the timeout interval.
  • if (a timeout is detected) --> re-transmit the frame.
  • -> There is a problem with re-transmitting the frame in this case...
  • -> B will receive a duplicate frame. Duplicate frames are NOT a good thing!
Q6 How can we recover from a duplicate frame?
  • Use ID's so that we can easily discard any duplicate frames.
  • Then resend an ACK back to A.
  • If we didn't resend an ACK back to A, it would keep sending duplicate frames forever...

In Q5, the length of the timeout interval is IMPORTANT!

Too long of a timeout interval does not cause protocol failure, rather it wastes bandwidth.

if (timeout interval is too long) {
    the sender will be waiting for too long
    therefore the data link will become idle
}

Too short of a timeout interval may cause the protocol to fail, which is a lot worse than just wasting bandwidth!

if (timeout interval is too short) {
    duplicate acknowledgements
    therefore the protocol fails
}

The solution to a short timeout interval, is to send the acknowledgement and the ID together (aka. Piggy-Backing)



Summary of 'When frames go wrong...'

  • All links are unreliable.
  • To recover from damaged frame --> use error check bits or ACK or NAK
  • To recover from lost frames or lost ACK or lost NAK --> use timeout.
  • To prevent duplicate frames or duplicate ACK or duplicate NAK --> use ID in frames, ACK's and NAK's
  • Use Piggy-Backing if the timeout interval is too short.
