Congestion Control

Congestion occurs when too many packets are present in the subnet.

If congestion is too high, routers begin loosing packets.

The routers loose packets because they are forced to throw away packets without inspecting them because there are no buffers left to store them.

Congestion may be caused by 2 things:

  • Network topology
  • Hardware

Congestion that is caused by network topology

Suppose we have a star topology network like the one shown below:


In such a network, it might be hard to predict data rate coming towards the central router.

This is because there are 3 routers that could all potentially send data at the same time.

This means the buffer could fill up very quickly, meaning packets will got lost quicker, resulting in congestion.

Most people will say "why not use a bigger buffer?"...

  • For a given router, if the time taken for a packet to travel from the back of the buffer to the front of the buffer is > timeout period, the router will resend the packet.
  • This results in duplicate packets.
  • Causing 'Congestion Collapse' (when a network cannot send or receive anymore packets)
  • So by using a bigger buffer, we managed to make congestion even worse!

Congestion tends to feed upon itself and become worse. Like a rotting corpse, you can't stop it from rotting when it's off.

Let's suppose we add 4 more routers to this network (Router 5, Router, 6, Router 7 and Router 8):


Router 7 sends data to Router 6, which sends data to Router 5, which sends data to Router 4.

So if Router 4 becomes congested routers 5, 6 and 7 will also become congested.

Those red numbers indicate how many packets each router wants to send (e.g. Router 2 wants to send 14 packets)

Router 4 will send all 10 packets at once, and will then wait for an ACK (assuming Router 4 uses a sliding window protocol)

Router 4 will save a copy of these 10 packets in its own buffer.

Buffer of Router 4 becomes congested.

Therefore, Routers 5, 6 and 7 become congested.

The congestion has spread through the network causing 'Backpressure' (when a router becomes congested, the congestion feeds upon itself and spreads in the network)

Congestion that is caused by hardware

Slow processors in routers can also cause congestion.

It is a common issue in networks that have machines of varying processing speeds (some slow, some fast, some normal)

Network engineers usually try to make the network speed < backbone speed

  • This is why the network uses twisted pair cables, and the backbone uses fibre optic cables.

    The following graph shows that the number of packets sent (aka. offered load), is proportional to the number of packets delivered (aka. goodput), atleast when the data rate is < max capacity.


    The max capacity of the network indicates how many packets can be in the network at one time.

    Open Loop vs. Closed loop

    Open Loop and Closed Loop are a collection of different solutions to control congestion.


    Open Loop solutions

    Wait for sender and receiver to sense that there is something wrong in the network.


    Closed Loop solutions

    Ask the network itself for some feedback on its status.

    1. Monitor the network to detect when and where congestion occurs, such as:
      • Percentage of all packets discarded for lack of buffer space.
      • Average queue lengths.
      • Number of packets that time out and re-transmitted.
      • Average packet delay.
      • Standard deviation of packet delay.
    2. Distribute this information to places where action can be taken:
      • Bad solution - create a choke packet that travels from the congested router to the sender. This is a bad solution because sending another packet through the network will add even more congestion!
      • Good solution - set the Explicit Congestion Notification (ECN) field in the IP header to 11.
    3. Adjust network operation to correct the problem:
      • there are 5 different approaches to adjusting the network due to congestion...

    Approaches to Congestion Control:

    These are the 5 approaches to dealing with congestion control:

    1. Network provisioning - very preventative
    2. Traffic-aware routing - preventive
    3. Admission control - somewhere in the middle
    4. Traffic throttling - reactive
    5. Load shedding - very reactive

    They are usually applied on different time scales to either prevent congestion or react to it once occurred.

    Let's look at each approach in detail:

    Network Provisioning

    • Basically asks if 'the network is going to be good?'
    • (ie. "Is it well matched to the traffic that it carries?")
    • (ie. "Are the routers fast enough?")

    Traffic-aware Routing:

    If the previous approach failed...

    • Compute best route using Dijkstra's Algorithm, taking into account the traffic.
    • Shift traffic away from congestion hotspot's by changing the shortest path weights.
    • Split traffic across multiple paths if necessary.
    • This approach is called 'Traffic-aware' because of the fact that it considers the network traffic in Dijkstra's Algorithm.

    Admission Control (traffic shaping):

    If the previous approach failed...

    • Allow new traffic to be loaded only if the network has sufficient capacity.
      • This reduces the overall traffic.
    • Make hosts transmit at a uniform rate (aka. do traffic shaping)
      • This approach is called 'Traffic-aware' because of the fact that it considers the network traffic in Dijkstra's Algorithm.
      • This is because one of the main causes of congestion is that traffic is often bursty. Hosts often don't transmit at consistent rates.
      • This approach is often called 'Traffic Shaping', and is considered part of this Admission Control approach.
      • There are 2 Traffic Shaping algorithms that I will cover shortly. These are the...
        • Leaky Bucket Algorithm
        • Token Bucket Algorithm

    Traffic Throttling (choke packets):

    If the previous approach failed...

    • Router creates a choke packet, and sends it back to the sender.
    • The choke packet is a way of telling all of the senders to 'SLOW DOWN!!! THERE'S TOO MUCH STUFF IN THE NETWORK'
    • After the sender receives the choke packet, the sender will reduce its TR in attempt to slow down.
    • This process may be repeated several times until the router is satisfied with the state of congestion in the network.
    • There are 2 ways of doing this:
      1. A choke packet that affects only the sender. Intermediate routers ignore the choke packet.
      2. A choke packet that affects each router it passes through.

    Load Shedding:

    If the previous approach failed...

    • The last resort if all other approaches failed is to make routers start dropping packets.
    • How you drop packets depends on the application being developed.
    • There are policies you should follow that dictate which packets to drop, how to drop them and when to drop them.
    • There are 2 ways of doing this:
      1. Wine approach - drop the newest packet. Used in applications that require reliability (e.g. email, viewing a webpage, downloading files)
      2. Milk approach - drop the oldest packet. Used in applications that require multimedia (e.g. video games, live-streams, video calling)

    Leaky Bucket Algorithm

    This is a Traffic Shaping algorithm.

    Let's begin with an analogy, using a literal bucket:


    • There is a tap which has water coming out of it.
    • The water falls into the bucket.
    • The bucket has a hole in it.
    • The water leaks out of the bucket at a constant rate, regardless of how fast or slow water is going into the bucket.
    • The leakage rate is dependent on the size of the hole.

    Now let's replicate this analogy in the context of a network:


    • The host (aka. sender) is equivalent to the tap.
    • The sender can send packets at any rate it likes.
      • In this diagram, it send 3 packets, then stops, then send 2 more packets.
      • This is labelled as 'Unregulated flow'.
    • The packets passes through the 'bucket'.
    • The packets leave the bucket at a constant rate.
      • You can see this constant rate after leaking out of the bucket.
      • This is labelled as 'Regulated flow'.
    • Depending on your application, you may or may not want a regulated flow.

    Token Bucket Algorithm

    This is another Traffic Shaping algorithm, an alternative to the Leaky Bucket Algorithm.

    Allows saving up tokens to the maximum size of the bucket, N.

    This means that bursts of up to N packets can be sent at once, allowing some burstiness in the output stream, and giving faster response to sudden bursts of input.

    This is great, but means the Token algorithm is vulnerable to large bursts if N is too big.

    A common solution to this it to use both the Token and Leaky algorithms (in that order).

    Sooo in summary, to get the smoothest traffic possible, put a Leaky bucket after the Token bucket.


    In Token Bucket, we are no longer restricted to a regulated flow of data.

    Instead, the flow of data can speed up or slow down depending on the size of the burst.

    It has the word 'token' in its name because each packet picks up a token on its way through the bucket. The packet can only progress into the network if it picked up one of these tokens.

    Now let's do an example:


    • The host (aka. sender) sends packets at any rate it likes.
    • The packets passes through the 'bucket'
    • One token (those black things in the bucket) is added to each packet as it passes through.
    • According to [diagram a] we have 5 packets, and 3 tokens in the bucket to begin with.
    • Therefore only 3 of the packets will be given tokens. Those 3 packets will pass through into the networks.
    • The remaining 2 packets will have to wait behind the bucket for more tokens to be given out.
